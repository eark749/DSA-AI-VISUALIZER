Overview
Two indexes, one retriever

Vector search finds semantically similar chunks. Keyword search finds exact matches. Hybrid retrieval uses both and re-ranks — best of both worlds.

🕸️
HNSW — approximate nearest neighbor
Hierarchical Navigable Small World. Multi-layer graph where each layer is a sparser subset of the previous. Navigate top-to-bottom for O(log N) search.
📑
Inverted index — exact keyword match
Maps term → list of document IDs. O(1) postings lookup. BM25 scores docs by term frequency and inverse document frequency. Used by Elasticsearch, Postgres FTS.
🔀
Hybrid retrieval
Run both. Merge results with Reciprocal Rank Fusion. Vector catches paraphrases; keyword catches exact product codes or names. Higher recall than either alone.

Python · Inverted index
from collections import defaultdict

class InvertedIndex:
    def __init__(self):
        # term → set of doc IDs
        self.index = defaultdict(set)

    def add_doc(self, doc_id, text):
        for token in tokenize(text):
            self.index[token].add(doc_id)

    def search(self, query):
        tokens = tokenize(query)
        sets = [
            self.index[t]
            for t in tokens
            if t in self.index
        ]
        if not sets:
            return []
        # Intersect posting lists
        return list(sets[0].intersection(*sets[1:]))

# Elasticsearch uses same concept
# at scale with BM25 scoring on top
idx = InvertedIndex()
idx.add_doc(1, "billing invoice charge")
idx.add_doc(2, "billing refund request")
results = idx.search("billing refund")
# → [2]  (only doc with both terms)

Key insights
What engineers get wrong
KEY INSIGHT — ef_search vs recall
HNSW's ef_search is the primary recall knob. Higher ef_search → more candidates evaluated → higher recall → higher latency. The Pareto frontier: ef_search=64 typically yields ~95% recall at ~2ms; ef_search=200 yields ~99.5% at ~8ms. Always benchmark your corpus — the numbers vary by dimension and distance metric.
KEY INSIGHT — Hybrid beats either alone
Semantic search (HNSW) excels at meaning; keyword search (inverted index) excels at exact terms. Reciprocal Rank Fusion merges both: score = 1/(k+rank). Neither method dominates across all query types. RAG benchmarks show hybrid consistently +5–15% NDCG over either alone.
Tuning parameters
HNSW index construction (one-time cost)
M (max connections per node): higher = better recall, more memory. Default 16.
ef_construction: beam width during insert. Higher = better graph quality, slower build. Default 200.
Rule: ef_construction ≥ M. For production, M=32, ef_construction=400 is common.
Inverted index: BM25 vs TF-IDF
TF-IDF: score = tf × log(N/df)
BM25: adds saturation (k1) and field-length norm (b). More accurate for long docs.
Elasticsearch uses BM25 by default (k1=1.2, b=0.75). For AI pipeline docs, lower b (~0.5) often wins.
Alternatives
Flat (brute-force) search
Compute exact cosine distance to every vector. O(N·d) per query. 100% recall by definition.
Use when: corpus <100K, recall must be exact, offline eval
IVF-PQ (Faiss)
Inverted file + product quantization. 10–50× memory compression vs HNSW. Slightly lower recall. Best for billion-scale.
Use when: corpus >100M, GPU available, memory-constrained
Sparse vectors (SPLADE)
Learned sparse representations. Stored in standard inverted index but with semantic expansion. Bridges keyword + semantic gap.
Use when: domain-specific jargon, no separate dense/sparse pipeline wanted
Real-world systems
Pinecone
Managed HNSW index. Hybrid search via metadata filters. Used by OpenAI, Notion, Zapier for RAG.
Weaviate
Open-source vector DB with built-in BM25 + HNSW hybrid. Native GraphQL API. Auto-embedding via modules.
Elasticsearch kNN
Dense vector field + BM25 in same index. RRF available natively since v8.8. Mature ops tooling.
pgvector
HNSW + IVFFlat inside Postgres. Zero new infra for teams already on Postgres. Supabase ships it by default.
